# Codex Artifact Contracts

Verified against the public OpenAI documentation available on 2026-07-13.
These are authoring baselines, not replacements for the current Codex runtime.
When static validation and runtime discovery disagree, preserve the artifact,
record the exact runtime version, and report the discrepancy.

## Cross-artifact invariants

- Prefer the smallest native surface: `AGENTS.md` for durable repository
  guidance, skills for reusable workflows, plugins for distribution, MCP/apps
  for live external capabilities, hooks for lifecycle enforcement, and rules
  for outside-sandbox command policy.
- Do not copy Claude or OpenCode slash-command trees into Codex. Built-in slash
  commands are product UI; reusable authored workflows in this marketplace are
  skills.
- Keep custom-agent TOML outside plugin bundles until OpenAI documents a plugin
  component for it.
- Keep credentials, OAuth tokens, cookies, live sessions, trust hashes, caches,
  and machine-specific runtime state out of generated artifacts.
- Static PASS means only that the artifact satisfies the documented shape.
  Runtime-only discovery, authentication, trust, transport, and behavior must
  be reported separately with the exact Codex version and isolated paths.

## Current moving surfaces

As of the verification date, permission profiles are Beta, execpolicy rules are
experimental, custom-agent authoring may evolve, and app-server plugin
list/read/install/uninstall methods are under development. Do not turn any of
those APIs into an unversioned compatibility promise. The plugin-hook manifest
field also has the validator/runtime mismatch described above; default
`hooks/hooks.json` is the interoperable builder convention.

The live Build plugins documentation accepts either a direct server map or a
wrapped `mcp_servers` object in plugin `.mcp.json`. The bundled
`$plugin-creator` validator present on the verification date instead requires a
legacy top-level `mcpServers` wrapper. Keep the documentation-correct direct
map as the generator default, record the validator discrepancy when it occurs,
and require pinned-runtime discovery rather than rewriting the artifact solely
to satisfy the stale validator.

Current capability toggles are centralized in the top-level `[features]` table
(and a few dedicated tables it fronts). Rather than freeze the fast-moving flag
list here, treat `scripts/check_codex_artifact.py` (which pins the exact
top-level config surface) and the built-in `$openai-docs` skill as the
authority for names and defaults. Powerful surfaces an author reaches this way
include Code Mode, agent memories, tool search, guardian approvals,
browser/computer use, and multi-agent orchestration; enable only what the task
needs and prove each at runtime. One stable consequence: web search is on by
default, so the `features.web_search*` toggles are legacy — the config checker
rejects them, mirroring Codex's own deprecation notice.
